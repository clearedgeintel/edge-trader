## 1. Project Overview & Vision

**edge-trader** is a fresh rewrite of an earlier Alpaca trading bot that had become spaghetti code after evolving from simple rule-based logic into a complex multi-agent LLM system.

**Core Philosophy**:
- **Data analysis first**, rules + scoring second.
- LLM used **only** for post-decision educational "Report Cards" and suggested config tweaks (never for trade decisions).
- Designed specifically for **small retail accounts** — capital preservation > aggressive returns.
- Transparent, auditable, and educational.
- Standalone and easy for others to spin up (Alpaca + Supabase + Docker/Railway).

**Target User**: Beginner to intermediate trader starting with $500–$1,000 who wants trustworthy automation they can understand.

**Success Criteria**:
- Positive expectancy on validated setups
- Controlled drawdowns
- High-quality signals via confluence scoring
- User can explain why any trade was taken using real data
- Easy standalone deployment

## 2. Architecture Overview (v1 — Pure TypeScript)

Single Node.js/TypeScript service for fast iteration and simplicity.

### Layered Architecture

```
Config & Scheduler
        ↓
Data Layer (Alpaca bars, snapshots, multi-TF caching)
        ↓
Analysis Layer (Indicators, features, regime detection, scoring)
        ↓
Signals / Strategy Layer (MomentumPullback + pluggable modules)
        ↓
Risk & Sizing Layer ← Absolute Veto Power
        ↓
Execution Layer (orders, brackets, reconciliation)
        ↓
Monitor Layer (exits, partials, trailing, attribution)
        ↓
Persistence (Supabase) + API + Observability + Report Cards
```

**Key Principles**:
- Risk module has **absolute veto** power.
- Analysis + Signals + Risk code is reused by both live trading and the backtester.
- Every signal produces a structured rationale object (used for storage, backtesting, and Report Cards).
- Heavy emphasis on testability (especially Risk and Analysis logic).
- Clear module boundaries for future evolution (possible Python analysis service in Phase 2).

## 4. Strategy: Momentum Pullback (Flagship)

### Core Idea
Catch healthy pullbacks in confirmed trending/momentum stocks or ETFs using multi-timeframe confluence.

### Daily Trend Filter (Higher Timeframe)
- Price above EMA(200) **or** positive slope on EMA(50/200)
- ADX(14) > 20–25 (trending, not choppy)
- Optional: Positive relative strength vs SPY/QQQ

### Lower Timeframe Entry (1H or 15m preferred)
- Pullback into value zone: EMA(9/21), lower Bollinger Band, or VWAP/support area
- **Volume decreases** on the pullback (healthy pause)
- RSI(14) in 35–55 range (resilient)
- Bullish confirmation candle or momentum resumption
- No major earnings/news within ±2 sessions

### Confluence Scoring (0–100)
Weighted combination of the above. Configurable minimum threshold (start at 65–75).

### Structured Rationale (produced by every signal)
```json
{
  "symbol": "AAPL",
  "score": 78,
  "reasons": ["Daily trend strong", "Pullback to EMA zone on lower volume", ...],
  "indicators": { "adx": 27.4, "rsi_15m": 42.1, ... },
  "regime": "trending_bull",
  "suggested_stop": 172.10,
  "suggested_target": 185.50,
  "risk_dollars": 4.85
}
```

This object is stored and used for backtesting attribution and Report Cards.

## 5. Risk Management & Position Sizing (Critical)

**Risk module has absolute veto power.** It is the highest priority module to implement.

### Recommended Small-Account Defaults

| Parameter                  | Default     | Notes                                      |
|---------------------------|-------------|--------------------------------------------|
| risk_per_trade_pct        | 0.005 (0.5%)| Very safe for small accounts               |
| max_portfolio_heat_pct    | 0.025       | Total capital at risk                      |
| max_concurrent_positions  | 3           | Keeps things manageable                    |
| atr_multiplier            | 1.8         | Adaptive volatility-based stops            |
| min_reward_risk           | 2.5         | Minimum 2.5:1 R:R                          |
| daily_loss_limit_pct      | 0.02        | Pause trading after bad day                |
| max_drawdown_pause_pct    | 0.10        | Circuit breaker from peak                  |
| correlation_threshold     | 0.70        | Avoid adding correlated risk               |
| min_avg_daily_volume      | 500000      | Liquidity filter                           |

### Position Sizing Formula
```ts
riskDollars = equity * riskPerTradePct
stopDistance = ATR * atrMultiplier
qty = riskDollars / stopDistance   // fractional supported
```

### Portfolio Guards (before every new trade)
- New trade risk + current open risk ≤ max heat
- Max positions not exceeded
- Correlation and sector limits respected
- Daily loss and drawdown limits not breached

**Veto is absolute** — no overrides allowed.

## 6. Backtesting & Validation

First-class citizen. Reuses the same Analysis + Signals + Risk modules as live trading for high fidelity.

**Capabilities**:
- Event-driven historical replay with realistic fills and slippage
- Regime-stratified results
- Full per-trade attribution (MAE, MFE, exit reason, regime, rationale)
- Walk-forward optimization + stress testing across bull/bear/chop periods

**Validation Path**:
Backtest → Paper trading agreement → Small live capital ramp (5–10% size first)

## 7. LLM / Report Card Layer (Post-Decision Only)

Triggered after signal generation or trade close.

**Input**: Structured rationale + recent performance stats + portfolio context

**Output**: Educational "Report Card" including:
- Thesis summary
- Risk snapshot
- Educational explanation
- Suggested config tweaks (e.g. "Consider raising ADX threshold in high VIX regimes")

Fully optional and toggleable. Uses cheap/fast model (e.g. Claude Haiku).
